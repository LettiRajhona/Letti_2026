Szoftverfejlesztési folyamat felépítése

A szoftvertechnológia – mint a tárgy nevében is szerepel – feladata szoftverek rendszerezett, felügyelt, minőségi fejlesztése, majd működtetése és karbantartása. A szoftver magába foglalja nemcsak az aktuális programokat, hanem minden konfigurációjukat és dokumentációjukat is. Megfelelő színvonalat kell biztosítania az elvárt funkciókat illetően, és úgynevezett minőségi mutatóknak (karbantarthatóság, biztonság, hatékonyság, használhatóság, ...) is eleget kell tennie.

Egy szoftvertechnológiai projekt során különféle szerepkörök oszthatók ki. A projektmenedzser felel a szoftver előírt minőségéért, miközben szervezi és irányítja a projekten dolgozó csapat munkáját. A termékgazda üzleti folyamatokban jártas személy. A programgazda a fejlesztés ütemezését látja el, követi és elosztja a feladatokat. Megkülönböztethetők még tervezők, fejlesztők és minőségbiztosító személyek is.

A szoftver életciklusát négy részre bonthatjuk. A specifikáció célja a feladatot megoldó szoftver funkcióinak tisztázása, a rendszerre és a fejlesztésre vonatkozó elvárások megadása. Részét képezi a megvalósíthatósági elemzés, amely a szoftver megvalósításának lehetséges körülményeit vizsgálja, továbbá azt, hogy a befektetés megtérül-e üzletileg. Tartalmazza a szükséges humán és egyéb erőforrásokat, a megvalósítás költségét, időtartamát, és a későbbi karbantartás eshetőségét.

Példa megvalósíthatósági tervre (vázlatosan):

 

1. Projekt áttekintése

Cél: Egy Windows alapú Safari Tycoon játék készítése, amelyben a felhasználó saját szafari parkot vezethet. 

Főbb funkciók:

  • Új játék kezdése.
  • Korábbi játék betöltése, mentés.
  • Megfelelő elemek, élőlények vásárlása, eladása.
  • Játékidő manipulálása.

2. Műszaki megvalósíthatóság

Technológiák:

  • Felhasználói felület és szoftver: Java Swing.
  • Adatbázis: SQLite.
  • JDBC használata adatbázis kapcsolathoz.

Előnyök:

  • Könnyen hordozhatóság, nem igényel külön szervert.
  • Alacsony erőforrásigény.
  • Üzemeltetni nem szükséges.
  • Esetleges hibajavításon felül nem kell biztosítani karbantartást.

Hátrányok:

  • Java Swingben nehezebb lehet modern grafikus felület használata.
  • Az SQLite nem ideális nagy adatmennyiséghez.

3. Operatív megvalósíthatóság

Humán erőforrások: 3 tervező, fejlesztő, tesztelő fejenként körülbelül 100 óra munkával.

Hardver követelmények: 3 fejlesztői számítógép, közepes hardverigénnyel.

Időkeret: MVP verzió elkészítéséig 13 hét áll rendelkezésre.

A megvalósítás időtartama körülbelül 300 óra. Ezen felül nem igényel extra költségeket.

 

A követelményelemzés során specifikációk megírására van szükség. A követelmények funkcionális és nem funkcionális specifikációra oszlanak. Míg az előbbi esetében a szoftver által nyújtott szolgáltatások, reakciók, viselkedések leírása történik, az utóbbi a szolgáltatásokra és viselkedésükre szab megszorításokat. Ide értendő a termékekkel szemben támasztott követelmények, menedzselési követelmények, külső követelmények.

Nézzünk egy példát először a nem funkcionális specifikációra.

 

1. Hatékonyság

  • A számítógép szempontjából a szoftver minimálisan erőforrásigényes.
  • A betöltési idő nem haladja meg a 1,5 másodpercet a szoftvert futtató eszközön, annak típusától függetlenül.
  • Gyorsan reagál bármilyen bevitelre alsó kategóriás számítógépeken is.

2. Megbízhatóság

  • Alapértelmezett használat mellett nem produkál hibajelenséget.
  • Nincs váratlan programleállás.

3. Biztonság

  • A lokális felhasználók jelszóval védett profilt hozhatnak létre, ahonnan korábbi játékállásaik elérhetőek.
  • A jelszavak titkosítottan kerülnek eltárolásra.

4. Hordozhatóság

  • Az alkalmazás kizárólag Windows környezetre készült, minden Windows operációs rendszert futtató eszközön működik.
  • Konténerizált deployment áll használatban Docker segítségével.

5. Felhasználhatóság

  • Az alkalmazás főleg asztali számítógépen és laptopon használható, de a megfelelő perifériák csatlakoztatásával egyéb eszközön is futtatható.
  • A fő funkciókhoz vezető útvonalak maximum 5 kattintásból elérhetők.

6. Fejlesztés

  • A használt programozási nyelv a Java 25, a megjelenítésért a Java Swing eszköztár felel.
  • Objetumorientált paradigma használat jellemző, amelyben az MVC modell megvalósul.
  • A szoftver forráskódja követi a Java kódolási szabályait és stílussal kapcsolatos konvencióit.
  • A git verziókezelés használatban áll.
  • A tesztelés automatizált unit testeket takar, triviális és nem triviális teszteseteket.
  • CI/CD ellenőrzés a gyors és biztonságos kiadás érdekében.
  • A kódhoz és rendszerhez kapcsolódó dokumentáció a GitLab Wiki felületén érhetőek el.

 

A funkcionális specifikációban azokat a tulajdonságokat és funkciókat taglaljuk, amelyeket a szoftver megvalósít. A Safari Tycoon esetén a használati eset diagram egyes eseteit valósítjuk meg.

 

Játékpálya

A játék pályája egy méretű négyzetrácson (mátrixon) alapul, amely meghatározott, -ös mérettel rendelkezik. Ezen a gridrácson kaphatnak helyet a különböző, megvásárolható és eladható tereptárgyak, mozgó és álló entitások, mint állatok és növények, továbbá a szafarit bejáró dzsipek a turistáikkal. A pálya bal felső sarkában található a szafari park bejárata, ahol a turisták addig várakoznak, amíg egy dzsip el nem szállítja őket egy véletlenszerű útvonalon a jobb alsó sarokban található kijáratig. Alapértelmezetten egy útvonal jelenik meg, amely a két kaput összeköti.

Főmenü

A játék indításakor be kell jelentkezni a lokális profilba. Ezután a főmenü tűnik fel, ahol több opció is található. 

  • Az Új játék gomb megnyomásakor kiválasztható a játék nehézségi szintje, és elindítható a szafari park.
  • A Játék betöltése gomb kattintásakor a profil korábbi mentett játékai találhatóak, egymás alatt felsorolással. Kiválasztás után elindítható az adott játékállás.
  • Az Információk pontban a játékkal kapcsolatos segédlet található. Tartalmazza a játék működésének legalapvetőbb szabályait, és megszorult helyzetben jó tanáccsal látja el a felhasználókat.
  • A Kilépés gomb bezárja a játék ablakát, és a futás befejeződik mentés nélkül.

A játék futás közben

A képernyő alsó részén, középen egy multifunkcionális sáv található, rajta az egyes boltokkal, a sebesség gombjaival és az elérhető tőke számával.

  • Boltpanel
    • Növénybolt. Növények (kaktusz, majomkenyérfa, nád...) vásárlására van lehetőség.
    • Állatbolt. Állatok (oroszlán, zebra, ...) vásárlására van lehetőség.
    • Járműbolt. Különböző dzsipek vásárlására van lehetőség.
    • Csempebolt. Vásárolhatók a dzsipekhez bejárható utak, valamint különböző terepi elemek (víz, hegy, föld, növényes-füves parcellák) is.
  • Sebességállító panel
    • Játékidő megállítása a gomb használatával.
    • Egyszeres sebesség beállítása a gomb használatával.
    • Kétszeres sebesség beállítása a gomb használatával.
    • Háromszoros sebesség beállítása a gomb használatával.

Az állatok görgővel kattintáskor chippelhetők, jobb klikkel pedig eladhatóak. Utóbbi érvényes a növényekre is. A minimap nagyítható, valamint kattintás esetén a meghatározott területre kerül a kamera. A játéktér mozgatható.

A játékterület parcellái

A választott extra feladatok, a térképgenerálás és terepi akadályok miatt szükséges különböző parcellákra bontani a játékteret. Megkülönböztethetünk magasság alapján magasabb és alacsonyabb területeket, vízi területeket, különleges mezőket és utakat is.

Vízi területek: ivásra alkalmas területek, amelyek környékén nád növény nő.

  • LAKE
  • BUBBLE LAKE
  • RIVER

Füves és homokos területek: normál területek, viszonylag sík terepen, az állatok mozgása normál sebességgel történik. Leginkább füvek, majomkenyérfák és kaktuszok nőnek rajtuk attól függően, hogy füves vagy homokos a parcella.

  • FIELD
  • WET FIELD
  • FLOWER (és más, 1-4 virágot tartalmazó FLOWER mezők)
  • SAND

Magasabb területek: olyan területek, amelyeknél az állatok mozgása korlátozott, leginkább sebességcsökkentés tekintetében. Minél magasabbra kerülnek, annál lassabban tudnak haladni.

  • HILL
  • MOUNTAIN

Különleges mezők: olyan mezők, amelyek megvásárolhatóak, de a szafari játékmenetében nem adnak hozzá semmihez. Alapvetően nem jelennek meg a pályán.

  • ICE
  • SNOW

Utak: az utakat csak a játék kezdete előtt lehet megvásárolni. Ezt egy felugró ablak jelzi a játékos számára. Elején és végén GATE, egy-egy be- és kilépési pont található.

  • ROAD: egyszerű út, a dzsipek mozgása ezen történik. Egy út automatikusan generálódik a map (0,0) és (75,75) koordinátájú parcellái között.
  • ARTIFICIAL ROAD: segédút. Eredetileg a vizes területek feltöltésére jött létre, hogy a későbbiekben az útépítés vizeken keresztül is megvalósulhasson.

Kilépő menü

A kilépő menü navigációja hasonló módon történik, mint a normál menüben. Több funkciót is ellát.

  • Vissza a játékba.
  • Hang ki- és bekapcsolása.
  • Aktuális játékállás mentése.
  • Vissza a főmenübe.
  • Kilépés a játékból.

A követelmény specifikáció során az elemzés által kapott információt átalakítjuk egységes alakba. Ez azt jelenti, hogy leírást készítünk a követelményekről, amely fejlesztői szinten releváns igazán. Ez a leírás lehet beszélt nyelven, formalizált leírással, formális matematikai leírással, leírónyelvekkel is. Számunkra a formalizált leírás a fontos: ide tartoznak a használati eset diagramok és a felhasználói történetek is.

A használati eset diagram a felhasználó és a rendszer kapcsolatát bontogatja. Három fő elemből áll. Az aktor az a felhasználó, aki a rendszert használja, és egyes funkciókkal találkozik. A funkció a rendszer egy szolgáltatása. Az előbbi kettőt úgynevezett reláció kapcsolja össze, amely meghatározza, milyen kapcsolatban áll a felhasználó az adott programegységgel.

Használati reláció esetén az aktor igénybe veheti a rendszer egy bizonyos funkcióját. Ez a kapcsolat kizárólag funkció és aktor között valósulhat meg.

Általánosításnak nevezzük azt az esetet, amikor az egyik funkció vagy aktor bővebb halmazt fed le, mint a másik. A nyíl vége arra a halmazra mutat, amelyik nagyobb.

A tartalmazás során az egyik funkció részét képezi egy másik funkciónak.

A kiterjesztés azt jelöli, hogy egy adott tevékenységet kiegészítünk egy másik funkcióval.

Előfeltétel reláció során egy funkciónak meg kell előznie egy másik funkciót.

A rákövetkezés akkor történhet, ha egy funkciót egy másik közvetlenül követ.

Példaként tekintsük meg a Safari játék egy lehetséges, bár nem túl bő használati eset diagramját.

 

Az egyes funkciók részletesebb kifejtéséhez felhasználói történetek készítése lehetséges. A felhasználó történetek a felhasználók által végzett tevékenységsorozatokat és azok egyes kimeneteit adják meg különböző szituációkban. Egy történetben meg kell határozni egy szerepkört, valamint egy célt és annak szükségét. Ilyen eset lehet például az, amikor oktatóként előadást szeretnénk tartani azért, hogy a hallgatók jobban megértsék a tananyagot.

A történetet rögzített kulcsszavakkal írhatjuk le, az As a ... I want to ... so that minta alapján. Egy történetre egy vagy több eset is felírható, amelyben adott környezetben (given) egy tevékenységet végzünk (when), és erre valamilyen hatást (then) várunk. Az eseteket úgy kell megfogalmazni, hogy az összes körülményt és lehetséges bemeneteket figyelembe vesszük. Különös figyelmet kell szentelni az edge case-ekre, hibalehetőségekre. Az egyes történetekkel a végleges alkalmazás funkcionális tesztesetei írhatóak le.

A Safari esetén a következő felhasználói történetek jöhetnek szóba:

AS A JÁTÉKOS
I WANT TO ELINDÍTANI A JÁTÉKOT
1 GIVEN új játék menüpontot választjuk
WHEN kiválasztjuk a nehézséget
THEN indítás gombra nyomunk, és a játék elindul
2 GIVEN játék betöltése menüpontot választjuk
WHEN kiválasztunk egy korábbi játékállást
THEN indítás gombra nyomunk, és a játék elindul

 

AS A JÁTÉKOS
I WANT TO VÁSÁROLNI A BOLTBAN
3 GIVEN megnyitjuk a boltpanelt
WHEN kiválasztjuk a vásárolandó objektum típusát
THEN vásárlás gombra nyomunk, és a kiválasztott helyen megjelenik

 

AS A JÁTÉKOS
I WANT TO MÓDOSÍTANI A JÁTÉKIDŐT
4 GIVEN megnyitjuk a sebességállító panelt
WHEN kiválasztjuk a játék megállítását
THEN a játék szünetel, amíg mást nem állítunk
5 GIVEN megnyitjuk a sebességállító panelt
WHEN kiválasztjuk a 2x-es sebességet
THEN a játék sebessége az eredeti sebességnél gyorsabbá válik

 

Objektumorientált tervezés

Az objektumorientált tervezés során a rendszert az objektumok mentén építjük fel, ahol az objektum több szerepkört is megvalósít. Egyrészt, egy absztrakció a valósághoz, biztosít egy elvárt funkcionalitást, továbbá az adat és működés által épülhet fel. A feladatok elvégzése során az objektumokat (osztályokat) be kell azonosítanunk a funkciók és adatok mentén, oly módon, hogy milyen építőelemeket tudnánk megfeleltetni a funkcióknak.

Az objektumelvű tervezés során öt alapelvet célszerű követni, amelyek a SOLID elvekként is ismertek.

  • Single Responsibility Principle (SRP): egy programegység csak egyvalamiért felelhet.
  • Open/Closed Principle (OCP): a programegységek nyitottak a kiterjesztésre, de zártak a módosításra.
  • Liskov Substitution Principle (LSP): az objektumok helyettesíthetőek altípusaik példányával.
  • Interface Segregation Principle (ISP): egy általános interfész helyett több kliens specifikus interfész.
  • Dependency Inversion Principle (DIP): az absztrakciótól függünk, nem a konkretizációtól.

Az osztálydiagramon többféle jelölés megtalálható az egyes objektumok kapcsolatát szimbolizálva.

  • Függőségi kapcsolatról beszélünk, amikor egy metódus epizódszerűen kapcsolatba kerül egy másik osztály objektumával.
public class Food {...}
public class Cat {
   public void feed(Food food) {...}
}

  • Az asszociáció olyan függőségi kapcsolat, amely hosszabb ideig fennáll az osztályok között. Ilyen eset, amikor adattagként megadunk egy másik osztály egy vagy több példányát. Jelölése folytonos vonallal történik, amely mindkét végén számérték található. A minimum érték az 1.
public class Student { ... }
public class Teacher {
   private List<Student> students;
   ...
}

  • Az aggregáció egy speciális asszociáció. Azt jelenti, hogy az egyik osztály objektumai részét képezik egy másik osztály objektumainak.
public class Engine {
   private int cylinders;
   public Engine(int cylinders) {
      this.cylinders = cylinders;
   }
}
public class Wheel {
   private String manufacturer;
   public Wheel(String manufacturer) {
      this.manufacturer = manufacturer;
   }
}
public class Car {
   private String name;
   private Engine engine;
   private List<Wheel> wheels;
   public Car(String name) {
      this.name = name;
      this.wheels = new ArrayList<>();
   }
   public void addWheel(Wheel wheel) {
      wheels.add(wheel);
   }
   public void addEngine(Engine engine) {
      this.engine = engine;
   }
   ...
}

  • A kompozíció egy speciális aggregáció, fizikai tartalmazást jelent osztályok között. Akkor használjuk, amikor nincs egyik objektum a másik nélkül – mint ember a testrészei nélkül. Telített rombusszal jelöljük.
public class Human {
   private Head head;
   private Body body;
   public Human() { ... }
   public void setHead(Head head) {
      this.head = head;
   }
   public void setBody(Body body) {
      this.body = body;
   }
   public void display() { ... }
}

public class Head {
   private String name;
   public Head(String name) {
      this.name = name;
   }
}

public class Body {
   private String name;
   public Body(String name) {
      this.name = name;
   }
}

  • Interfész megvalósítás esetén szaggatott teli nyilat használunk, amely az interfész felé mutat. 
  • Osztályból származtatás esetén sima teli nyilat használunk, ez esetben a nyíl az ősosztályra mutat.

Nézzünk most példát arra, hogyan építhető fel egy kezdetleges osztálydiagram. Ez persze bővíthető egyéb osztályokkal, amiket alkalmazunk.

A szoftver különálló egységekből épülhet fel. A szoftver komponens egy adott funkcionalitásért felelő, fizikailag elkülönülő része a rendszernek. Önállóan telepíthető, belső működése rejtett, és olyan funkcionalitást is szolgáltathat, amelyet más komponensek használnak fel. Egy szoftverrendszerben számos komponenst ismerhetünk: mobil és asztali alkalmazások, weblapok, webszolgáltatás, adatbázis... Természetesen egy program is felbontható komponensekre.
